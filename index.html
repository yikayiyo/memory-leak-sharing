基于 TiantianApp 文本点读场景

📋 背景

场景描述

文本点读题型，用户录音完成后启动播放原音，但在音频未加载完成时立即切题，导致内存泄漏。

技术栈

React Native + TypeScript + react-native-sound

影响范围

所有使用 AudioManager 的音频播放场景

🔍 问题复现流程

  1. 进入文本点读题
  2. 点击录音按钮，完成录音
  3. 点击"播放原音"按钮，启动音频播放
  4. 音频正在加载中（未开始播放）
  5. 立即点击"下一题"，切到其他题目

🐛 根本原因分析

代码路径分析

第一步：启动播放原音 
// 文件：AudioManager.ts
public startPlayFile(
  config: ({ path: string } & AudioPlayOptions) | string,
  callback?: AudioPlayCallback
): string | undefined {
  let soundId: string = Date.now() + '_' + Math.random();

  // 初始化状态
  this.playStatusMap[soundId] = PlayStatus.PLAY;
  this.soundStatusMap[soundId] = SoundStatus.WAIT_FOR_READY; // ⚠️ 等待加载
  this.playerMap[soundId] = new Sound(path); // ⚠️ 创建 Sound 对象

  return soundId;
}

第四步：AudioManager 停止逻辑 
// 文件：AudioManager.ts
public async stopPlay(soundId: string): Promise<void> {
  // ⚠️ 音频未加载完成，直接返回
  if (this.soundStatusMap[soundId] === SoundStatus.WAIT_FOR_READY) {
    this.soundStatusMap[soundId] = SoundStatus.WAIT_FOR_STOP;
    return resolve(); // ❌ 没有清理资源！
  }

  // ... 正常释放逻辑
}

❌ 核心问题：WAIT_FOR_READY 状态下没有调用 releaseSoundPlayer()

💥 内存泄漏形成过程

T1: startPlayFile("原音.mp3")
  • playerMap["12345"] = Sound对象
  • soundStatusMap["12345"] = WAIT_FOR_READY
  • playStatusMap["12345"] = PLAY
T2: 用户点击"下一题"

onMainQuestionChange() 被调用

T3: stemRef.stop()

audioRef.stop() 停止播放

T4: 切题完成
  • ❌ playerMap["12345"] 还在内存中
  • ❌ soundStatusMap["12345"] = WAIT_FOR_STOP
  • ❌ Sound 对象没有调用 .release()
  • ❌ Native 层的音频资源未释放

内存泄漏验证

使用 Chrome DevTools Memory 面板检测：

  • 初始内存：50MB
  • 播放原音 → 切题 → 55MB (+5MB)
  • 播放原音 → 切题 → 60MB (+5MB)
  • 播放原音 → 切题 → 65MB (+5MB)
  • 播放原音 → 切题 → 70MB (+5MB)

结论：每次切题后，内存增加 ~5MB，且无法回收！

🔧 解决方案

方案1：修复 AudioManager.stopPlay（推荐） 
// 问题代码
if (this.soundStatusMap[soundId] === SoundStatus.WAIT_FOR_READY) {
  return resolve(); // ❌ 没有清理
}

// 修复代码
if (this.soundStatusMap[soundId] === SoundStatus.WAIT_FOR_READY) {
  // ✅ 即使音频未加载完成，也要释放资源
  this.releaseSoundPlayer(soundId);
  return resolve();
}
方案2：切题时清理所有播放器 
// 文件：BaseQuestionPage.tsx
public onMainQuestionChange(mainQuestionIndex: number) {
  if (this.props.mainQuestionIndex != mainQuestionIndex) {
    // ✅ 停止 stemRef
    this.stemRef?.current?.stop?.();

    // ✅ 停止 AudioManager 中所有播放器
    AudioManager.GetInstance().stopAllPlayer(); // 新增
  }
}
方案3：StemView 停止时清理 AudioManager 
// 文件：StemView.tsx
public stop() {
  const mediaInfo = this.mediaRefList[this.state.curMediaIndex];
  if (!mediaInfo) return;

  // ✅ 停止 AudioManager 中所有播放器
  AudioManager.GetInstance().stopAllPlayer(); // 新增

  if (mediaInfo.mediaItem.type === MediaItemType.AUDIO) {
    mediaRef.audioRef?.current?.stop();
    return;
  }
}

✅ 最佳实践总结

📌 音频播放器管理原则

✅ DO:

  • 每个播放实例必须有唯一的 soundId
  • 播放完成后必须调用 release()
  • 切换页面时必须停止所有播放器
  • 未加载完成的播放器也要清理资源

❌ DON'T:

  • 只停止 UI 层面的播放
  • 认为异步加载的音频会自动清理
  • 在 WAIT_FOR_READY 状态下直接返回
  • 依赖垃圾回收自动清理 Native 资源

📌 React Native 生命周期清理

// ✅ 正确：componentWillUnmount 中清理
class MyPage extends React.Component {
  componentWillUnmount() {
    // 停止所有播放器
    AudioManager.GetInstance().stopAllPlayer();

    // 清理监听器
    AudioManager.GetInstance().removePlayStatusListener(cb);

    // 清理定时器
    clearTimeout(this.nextClickTimeout);
  }
}

📌 状态机处理异步操作

// ✅ 正确：所有状态分支都要清理资源
public async stopPlay(soundId: string): Promise<void> {
  if (this.soundStatusMap[soundId] === SoundStatus.WAIT_FOR_READY) {
    // ✅ 未加载完成：也要清理
    this.releaseSoundPlayer(soundId);
    return resolve();
  }

  // 其他状态...
  this.releaseSoundPlayer(soundId);
  return resolve();
}

📌 组件卸载时的完整清理

class BaseQuestionPage extends React.Component {
  componentWillUnmount() {
    // 1. 停止当前播放的音频
    if (this.playSoundId) {
      this.audioManager.stopPlay(this.playSoundId);
    }

    // 2. 停止所有播放器（兜底）
    this.audioManager.stopAllPlayer();

    // 3. 清理定时器
    clearTimeout(this.nextClickTimeout);

    // 4. 清理监听器
    this.audioManager.removePlayStatusListener(cb);
  }
}

🎯 总结

核心问题

  1. 状态机不完整： AudioManager.stopPlay() 在 WAIT_FOR_READY 状态下没有清理资源
  2. 生命周期缺失： 切题时只停止了 UI 层面的播放，没有清理 AudioManager
  3. 资源未释放： Native 层的 Sound 对象没有调用 release()
优先级 方案 影响范围 难度
🔥 P0 修复 AudioManager.stopPlay 所有音频播放 低
🔥 P0 切题时调用 stopAllPlayer 切题场景 低
📌 P1 StemView 停止时清理 AudioManager 题干音频 低
📌 P2 添加内存泄漏监控日志 调试 低
